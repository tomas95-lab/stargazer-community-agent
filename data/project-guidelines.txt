Project Overview
The purpose of this project is to create a database of challenging issue requests for a
repository. This database will be used to benchmark and improve the ability of coding
agents to implement changes in a codebase in response to user prompts.
To achieve this, you will build a standard project setup around an assigned repository. You
will then create a challenging prompt requesting changes to the codebase. You will check
that the model fails to respond adequately to your prompt, write your own solution to the
issue, create unit tests to verify your solution, and compose a rubric that can be used to
grade how successful the agent was in responding to your issue. When you are finished,
you will prepare the required patches, scripts, and Docker files so the whole workflow can
be reproduced and validated later.

Onboarding &
Installations
Before tasking for the first time, you must complete the project

adds all of the requested functionality without breaking any
previously existing functionality.

Scripts

on the task is reproducible.

all of the unit tests that you wrote.

grade how good or bad a response to your prompt is.
Evaluation Script
Before submission, you will run an evaluation script on your task.
The suite checks the quality and alignment of your prompt, tests,
rubrics, gold patch, Dockerfiles, and granularity.

● Base Dockerfile (base.Dockerfile): This Dockerfile sets up the environment by
creating /app, cloning the repository into it, and installing any required system or
project dependencies.
● Instance Dockerfile (instance.Dockerfile): This Dockerfile extends the base image
and applies patches/basetoinstance.patch to /app. The build logic must handle
the patch file, whether it is populated (this will be the case in Bug Injection tasks) or
empty (this will be the case in all other tasks). In the case of Bug injection, the patch
serves to transition the codebase into the designated problem state for evaluation.
3. Scripts
● The scripts/ directory contains executable shell scripts (run_script.sh,
parse_results.sh) that are mounted into the container under /app/scripts.
These scripts are responsible for applying additional patches, running tests, and
parsing results.
4. Patches
● The patches/ directory defines all transformations applied to the repository at
/app.
○ basetoinstance.patch modifies the cloned repository to represent the
problem scenario.
○ test_patch.patch introduces tests that should fail before your patch is
applied.
○ gold_patch.patch introduces your solution to the issue, which should pass all
the unit tests.
5. JSONFormatted Tests Results
● The test_results.json output from parse_results.sh

Prompt and Rubric Alignment:
At the end of the task, you will write a rubric that can be used to grade at a granular level
how well the model did in solving your issue. The prompt and the rubric need to be
aligned on their level of specificity. This means:
● Prompts must explicitly mention filenames, paths, functions, or modules if the rubric
criteria reference them. If the rubric lists specific paths or APIs, the prompt must also
contain those exact references.
● It is acceptable to make the prompt unrealistically specific if that is necessary to
align with the level of granularity in the rubric and guarantee that the rubric criteria
unambiguously evaluate the solution
How to build a good prompt:
The prompt is the foundation of the task. Its quality determines the quality of every
component built on top of it (test patch, golden patch, and rubrics). Treat this step
carefully.
The goal of the prompt is to realistically simulate an issue a developer would encounter
while modifying or improving a repository. Write from that perspective: a real request,
grounded in the repo, phrased the way a developer would actually phrase it.
Follow these principles:
● Write it like a real GitHub issue. The prompt should read like something a
developer would actually open in a repo focused on one problem or request. Try
not to stack multiple unrelated changes, pile on extra features, or ask for several
different things in the same prompt. One issue, one ask. If you find yourself writing
"and also," stop and split the idea or cut it.
● Be clear and concise. State the problem directly. Difficulty should come from the
realism of the problem, not from artificial complexity or volume.
● Eliminate major ambiguity. Before finalizing, reread the prompt and ask whether
two competent developers would interpret the requirements the same way. If not,
tighten the wording.
● Use explicit requirements. Every requirement must be concrete enough that it can
be verified by a test and satisfied by the gold path solution. Avoid vague language
("make it better," "improve performance") unless the success criteria are clearly
implied by context.

Step 7 Rubric Creation (return to TOC
A rubric is a set of criteria that can be used to grade objectively and granularly how good
a response is to a given prompt. You will create a series of criteria that exhaustively cover
all the features of an ideal response to the prompt that you created. These criteria should
cover all the features of a good response without being so restrictive that they exclude
other possible good solutions. Therefore, you can start with your Golden Solution as a
model for the rubric criteria, but make sure that you donʼt write them in such a way that
other possible good implementations are left out.
Guidelines (return to TOC
● For tips, tricks, and examples, consult the Tips and Tricks for Creating Rubrics
section in the Appendix!
● You may write as many criteria as needed.
● Each criterion should match one (or more) parts of the prompt. If your criterion does
not have its counterpart in the prompt, it is most likely that it is over-specific or
ambiguous.
● There may be zero, one, or multiple criteria per quality dimension.
● You will assign weights to each criterion based on how important it is to the issue
that you created.
